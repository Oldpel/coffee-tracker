## 项目概述

### 1.1 项目定位

**咖啡豆养豆记录与社群交流平台** 是一款面向社群咖啡爱好者的专业工具应用，旨在帮助用户：

- 📊 **系统记录** 咖啡豆的养豆进度和冲煮数据
- 🤖 **获取建议** 基于豆子属性的个性化冲煮曲线推荐
- 💬 **社群交流** 与其他咖啡爱好者分享经验和讨论

### 1.2 目标用户

- **主要用户**: 社群咖啡爱好者、咖啡烘焙师、咖啡馆经营者
- **用户规模**: 初期 1000-5000 活跃用户
- **用户特征**: 对咖啡有深度热情，愿意分享和学习

### 1.3 核心价值主张

| 价值维度 | 描述 |
|---------|------|
| **数据管理** | 完整记录每颗豆子的养豆周期和冲煮参数，形成个人数据库 |
| **智能建议** | AI 根据豆子的烘焙度、处理法、产地生成个性化冲煮方案 |
| **社群互动** | 安全的讨论交流空间，屏蔽非法内容，促进知识分享 |
| **稳定可靠** | 高性能架构确保数据安全和平台稳定性 |

## 数据库设计

### 4.1 数据模型

#### 4.1.1 用户表 (users)

```sql
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  openId VARCHAR(64) UNIQUE NOT NULL,      -- Manus OAuth ID
  name TEXT,                                -- 用户名
  email VARCHAR(320),                       -- 邮箱
  loginMethod VARCHAR(64),                  -- 登录方式
  role ENUM('user', 'admin') DEFAULT 'user', -- 用户角色
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  lastSignedIn TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**字段说明**:
- `openId`: Manus OAuth 唯一标识，用于用户识别
- `role`: 用户角色，admin 可以进行内容审核
- `lastSignedIn`: 最后登录时间，用于统计活跃用户

#### 4.1.2 咖啡豆表 (coffee_beans)

```sql
CREATE TABLE coffee_beans (
  id INT PRIMARY KEY AUTO_INCREMENT,
  userId INT NOT NULL,                      -- 所有者用户 ID
  name VARCHAR(255) NOT NULL,               -- 豆子名称
  origin VARCHAR(255),                      -- 产地
  processingMethod VARCHAR(100),            -- 处理法
  roastLevel VARCHAR(50),                   -- 烘焙度
  purchaseDate TIMESTAMP NOT NULL,          -- 入手日期
  notes TEXT,                               -- 备注
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (userId) REFERENCES users(id) ON DELETE CASCADE,
  INDEX (userId, createdAt)
);
```

**字段说明**:
- `userId`: 数据隔离关键字段，确保用户只能访问自己的豆子
- `purchaseDate`: 用于计算养豆状态
- `processingMethod`, `roastLevel`, `origin`: AI 建议的输入参数

#### 4.1.3 冲煮记录表 (brewing_records)

```sql
CREATE TABLE brewing_records (
  id INT PRIMARY KEY AUTO_INCREMENT,
  beanId INT NOT NULL,                      -- 关联豆子 ID
  userId INT NOT NULL,                      -- 所有者用户 ID
  brewDate TIMESTAMP NOT NULL,              -- 冲煮日期
  brewMethod VARCHAR(100),                  -- 冲煮方式
  waterTemperature INT,                     -- 水温（℃）
  grindSize VARCHAR(100),                   -- 研磨度
  coffeeAmount DECIMAL(5,2),                -- 咖啡粉量（克）
  waterAmount DECIMAL(5,2),                 -- 水量（毫升）
  brewTime INT,                             -- 冲煮时间（秒）
  tasteRating INT,                          -- 口感评分（1-10）
  notes TEXT,                               -- 备注
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (beanId) REFERENCES coffee_beans(id) ON DELETE CASCADE,
  FOREIGN KEY (userId) REFERENCES users(id) ON DELETE CASCADE,
  INDEX (userId, brewDate),
  INDEX (beanId, brewDate)
);
```

**字段说明**:
- `userId`: 数据隔离，冗余存储以优化查询性能
- `beanId`: 关联到具体豆子
- `brewDate`: 用于排序和统计

#### 4.1.4 社群讨论表 (community_posts) - 新增

```sql
CREATE TABLE community_posts (
  id INT PRIMARY KEY AUTO_INCREMENT,
  userId INT NOT NULL,                      -- 发布者用户 ID
  title VARCHAR(255) NOT NULL,              -- 讨论标题
  content LONGTEXT NOT NULL,                -- 讨论内容
  category VARCHAR(50),                     -- 分类（豆子分享/冲煮技巧/设备评测/其他）
  relatedBeanId INT,                        -- 关联豆子 ID（可选）
  status ENUM('published', 'pending', 'rejected', 'deleted') DEFAULT 'pending', -- 审核状态
  viewCount INT DEFAULT 0,                  -- 浏览次数
  likeCount INT DEFAULT 0,                  -- 点赞数
  commentCount INT DEFAULT 0,               -- 评论数
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (userId) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (relatedBeanId) REFERENCES coffee_beans(id) ON DELETE SET NULL,
  INDEX (userId, createdAt),
  INDEX (status, createdAt),
  INDEX (category, createdAt)
);
```

**字段说明**:
- `status`: 审核状态，pending 表示待审核
- `category`: 讨论分类，便于用户筛选
- `relatedBeanId`: 可选关联豆子，用于上下文关联

#### 4.1.5 社群评论表 (community_comments) - 新增（两级平铺结构）

```sql
CREATE TABLE community_comments (
  id INT PRIMARY KEY AUTO_INCREMENT,
  postId INT NOT NULL,                      -- 关联讨论 ID
  userId INT NOT NULL,                      -- 评论者用户 ID
  content LONGTEXT NOT NULL,                -- 评论内容
  rootCommentId INT,                        -- 根评论 ID（一级评论此字段为空）
  replyToUserId INT,                        -- 被回复者 ID（用于提醒和前端展示）
  status ENUM('published', 'pending', 'rejected') DEFAULT 'pending', -- 审核状态
  isDeleted BOOLEAN DEFAULT FALSE,          -- 软删除标记
  likeCount INT DEFAULT 0,                  -- 点赞数
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (postId) REFERENCES community_posts(id),
  FOREIGN KEY (userId) REFERENCES users(id),
  FOREIGN KEY (rootCommentId) REFERENCES community_comments(id) ON DELETE CASCADE,
  INDEX (postId, rootCommentId, createdAt)  -- 优化两级评论查询性能
);
```

**设计优化**:
- 使用 `rootCommentId` 代替 `parentCommentId`，将评论扩展为两级平铺结构
- 一级评论: `rootCommentId` 为 NULL
- 二级评论: `rootCommentId` 指向一级评论 ID
- 提高查询性能，避免深度递归

#### 4.1.6 内容审核表 (content_moderation) - 新增

```sql
CREATE TABLE content_moderation (
  id INT PRIMARY KEY AUTO_INCREMENT,
  contentType ENUM('post', 'comment') NOT NULL, -- 内容类型
  contentId INT NOT NULL,                   -- 内容 ID
  userId INT NOT NULL,                      -- 内容所有者
  reportedBy INT,                           -- 举报者 ID
  reason VARCHAR(255),                      -- 举报原因
  status ENUM('pending', 'approved', 'rejected', 'resolved') DEFAULT 'pending', -- 审核状态
  reviewedBy INT,                           -- 审核者 ID
  reviewNotes TEXT,                         -- 审核备注
  action ENUM('none', 'warning', 'delete', 'ban') DEFAULT 'none', -- 处理方式
  isDeleted BOOLEAN DEFAULT FALSE,          -- 软删除标记
  createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (userId) REFERENCES users(id),
  FOREIGN KEY (reportedBy) REFERENCES users(id) ON DELETE SET NULL,
  FOREIGN KEY (reviewedBy) REFERENCES users(id) ON DELETE SET NULL,
  INDEX (status, createdAt),
  INDEX (userId)
);
```

### 4.2 数据一致性与软删除策略

**软删除优势**:
- 不物理删除数据，保持历史记录
- 便于数据审计和恢复
- 避免级联删除带来的数据丢失

**实现方式**:
```typescript
// 查询时自动筛选未删除数据
const beans = await db.select()
  .from(coffeeBeansTable)
  .where(
    and(
      eq(coffeeBeansTable.userId, userId),
      eq(coffeeBeansTable.isDeleted, false)  // [Security] 强制筛选
    )
  );

// 删除时更新标记，不物理删除
const result = await db.update(coffeeBeansTable)
  .set({ isDeleted: true, updatedAt: new Date() })
  .where(
    and(
      eq(coffeeBeansTable.id, beanId),
      eq(coffeeBeansTable.userId, ctx.user.id)  // [Security] 校验数据归属
    )
  );
```

### 4.4 数据隔离策略

**关键原则**: 每个用户只能访问自己的数据

#### 实现方式

1. **用户 ID 过滤**: 所有查询都必须包含 `userId` 条件
   ```typescript
   // 示例：获取用户的豆子
   const beans = await db.select()
     .from(coffeeBeansTable)
     .where(eq(coffeeBeansTable.userId, userId));
   ```

2. **冗余存储**: 在 `brewing_records` 中冗余存储 `userId`
   ```typescript
   // 即使通过 beanId 查询，也要验证 userId
   const record = await db.select()
     .from(brewingRecordsTable)
     .where(
       and(
         eq(brewingRecordsTable.id, recordId),
         eq(brewingRecordsTable.userId, userId)
      )
    );
   ```

3. **级联删除**: 用户删除时自动删除其所有数据
   ```sql
   FOREIGN KEY (userId) REFERENCES users(id) ON DELETE CASCADE
   ```

#### 8.2.5 全局限流中间件 [Security]

```typescript
// server/_core/rateLimit.ts
import rateLimit from 'express-rate-limit';

// 通用限流器：每个 IP 每分钟最多 60 请求
export const globalLimiter = rateLimit({
  windowMs: 60 * 1000,        // 1 分钟
  max: 60,                     // 最多 60 个请求
  message: 'Too many requests, please try again later',
  standardHeaders: true,       // 返回 RateLimit-* 头
  legacyHeaders: false,        // 禁用 X-RateLimit-* 头
  skip: (req) => {
    // 跳过健康检查端点
    return req.path === '/health';
  },
});

// 严格限流器：API 端点每个用户每分钟最多 20 请求
export const apiLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 20,
  keyGenerator: (req) => {
    // 如果用户已认证，使用用户 ID；否则使用 IP
    return req.user?.id?.toString() || req.ip || 'anonymous';
  },
  message: 'API rate limit exceeded',
});

// 内容发布限流器：防止刷屏
export const contentLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 3,                      // 每分钟最多 3 个新帖子
  keyGenerator: (req) => req.user?.id?.toString() || req.ip,
  message: 'You are posting too frequently, please wait',
});

// 在 Express 应用中应用
app.use(globalLimiter);                          // 全局限流
app.use('/api/trpc', apiLimiter);                // API 限流
app.post('/api/trpc/community.createPost', contentLimiter); // 内容发布限流
```

**限流策略**：
- **全局限流**：保护服务器不被滥用
- **API 限流**：防止单个用户过度调用
- **内容限流**：防止刷屏和垃圾内容

#### 8.2.7 游标分页实现

```typescript
// server/db.ts
export interface CursorPaginationParams {
  cursor?: number;  // 上一页最后一条记录的 ID
  limit: number;    // 每页数量
}

export interface CursorPaginationResult<T> {
  items: T[];
  nextCursor?: number;  // 下一页游标
  hasMore: boolean;
}

/**
 * 游标分页查询
 * 相比 offset/limit，游标分页性能更稳定
 */
export async function getCommunityPostsWithCursor(
  userId: number,
  params: CursorPaginationParams
): Promise<CursorPaginationResult<CommunityPost>> {
  const { cursor, limit } = params;

  // 构建查询条件
  let query = db.select()
    .from(communityPostsTable)
    .where(
      and(
        eq(communityPostsTable.status, 'published'),
        eq(communityPostsTable.isDeleted, false)
      )
    )
    .orderBy(desc(communityPostsTable.createdAt));

  // 如果有游标，从游标位置开始
  if (cursor) {
    query = query.where(lt(communityPostsTable.id, cursor));
  }

  // 获取 limit + 1 条记录，用于判断是否有下一页
  const items = await query.limit(limit + 1);

  // 判断是否有下一页
  const hasMore = items.length > limit;
  const result = items.slice(0, limit);

  return {
    items: result,
    nextCursor: result.length > 0 ? result[result.length - 1].id : undefined,
    hasMore,
  };
}
```

**游标分页优势**：
- 性能稳定，不依赖总记录数
- 适合无限滚动加载
- 处理数据变化（新增、删除）时更准确
